**四、 遇到的问题与困难**

*   **技术问题：**
    *   SQLite3 C API 直接操作，参数绑定步骤繁琐（需手动执行 prepare → bind → step → finalize 流程），且错误信息不够直观，排查问题效率较低
    *   **已尝试的解决方案：** 封装统一的 DAO 层，将标准流程模板化；每次操作后检查返回码，通过 `sqlite3_errmsg()` 获取详细错误信息并记录日志便于排查
    *   **需要帮助或待解决：** 无
    *   vcpkg 包管理器存在依赖版本冲突和编译失败问题
    *   **已尝试的解决方案：** 固定 vcpkg 的 baseline 版本，使用 `vcpkg-configuration.json` 锁定依赖版本；统一采用 `x64-mingw-dynamic` triplet 进行编译
    *   **需要帮助或待解决：** Windows 控制台中文输出出现乱码问题
    *   **已尝试的解决方案：** 在入口处添加 `SetConsoleOutputCP(65001)` 设置 UTF-8 代码页，HTTP 响应头同步设置 `charset=utf-8`
    *   **需要帮助或待解决：** Crow 微框架 URL 查询参数获取方式与主流框架（如 Express、Flask）存在差异，文档较少
    *   **已尝试的解决方案：** 阅读框架官方源码和示例，使用 `req.url_params.get()` 获取查询参数，注意返回值为 C 字符串指针需判空后转换
    *   **需要帮助或待解决：** 股票交易中首次买入（INSERT 新建持仓）与加仓（UPDATE 累加数量）两种场景的逻辑边界处理
    *   **已尝试的解决方案：** 买入时先 SELECT 查询是否已有持仓，存在则 UPDATE 累加，不存在则 INSERT 新建；卖出时 Controller 层先校验持仓数量，DAO 层执行 UPDATE 减仓，形成双重保障
*   **非技术问题：**
    *   本周需同时学习多个新工具和框架（Qt、DB Browser、Apifox、vcpkg、Crow），学习成本较高，导致实际编码时间被压缩
    *   **已采取的措施：** 优先通过官方文档和示例快速上手核心功能，边学边用；后续计划在项目启动前提前预留工具链学习时间，减少开发过程中的学习中断

*   **对项目或开发流程的思考：**
    *   三层 MVC 架构虽然在初期会增加代码量，但在接口数量增长到 10 个以上时，其分层优势开始显现——修改 DAO 层逻辑不影响 Controller 层，新增接口只需组合现有 DAO 方法
    *   C++ 生态的包管理和构建工具链相比 Python/Node.js 等语言更加复杂，前期在环境搭建上投入的时间虽然较多，但一套稳定的构建配置能为整个开发周期提供可靠的编译基础
    *   在动手编码前先通过 DB Browser 可视化设计数据库表结构，能有效减少后期因表结构不合理导致的反工，这种"先设计再实现"的方式值得在后续模块开发中坚持
